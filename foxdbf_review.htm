CDX CDX Index Parsing & Navigation

CDX-1 Integer overflow in leaf-page record number reconstruction Real Bug

In cdx.java → readPage(), the LEAF branch assembles a 4-byte record number from individual int variables and then shifts:

int k3 = bd[z+3]; if(k3<0) k3+=0x100;
long r = (k3<<24) | (k2<<16) | (k1<<8) | k0;   // ← overflow here

When k3 ≥ 0x80 (i.e., record number ≥ 2,147,483 per byte layout), k3<<24 produces a negative int because the shift happens in int arithmetic before assignment to long r. Java then sign-extends the negative int to 64 bits, corrupting the high word.

Correct fix — cast to long before shifting:

long r = ((long)k3<<24) | ((long)k2<<16) | ((long)k1<<8) | (long)k0;

In practice VFP databases rarely reach 8M+ records, so this is a latent but real bug. The same pattern appears correctly elsewhere using readNumber() — this one spot uses the manual 4-byte construction instead.

Notes on ToDo: Fixed
CDX-2 kb decode buffer may be undersized for large page fills Low Risk (VFP)

In cdx.java → prepArrs():

kb = new byte[1024<<3];   // 8 192 bytes

Each key entry written into kb consumes key_len + 6 bytes (key + 4 recno + 1 dup + 1 trail). With the buffer set for 300 keys at length 100, the decoded output could reach 300 × 106 = 31 800 bytes — well over 8 192. In actual VFP a 512-byte page with 488 bytes of data limits real key counts far below that, but the relationship between prepArrs()'s magic numbers is invisible — a future maintainer enlarging the buffer count could silently cause ArrayIndexOutOfBoundsException at decode time.

Add a comment linking these constants, or size kb as key_cnt * (key_Clen + 6).

Notes on ToDo: Nope. 8Kb for a leaf should be enough. Or something is "too much".
Reindex the database in FoxPro more often and avoid too long index keys. Then no leaf overflow will happen.
CDX-3 pgC second dimension fixed at 100 — shadows field name Minor Risk

In prepArrs() in both cdx.java and idx.java:

int key_cnt = 300;       // local — shadows this.key_cnt!
int key_Clen = 100;
pgC = new byte[key_cnt][key_Clen];

The local variable key_cnt silently shadows the instance field this.key_cnt. Any compiler warning about shadowed fields would fire here. More importantly, key_Clen = 100 matches VFP's key length ceiling, so this is safe for the stated use case, but the magic number should be a named constant or derived from the header's key_len field to make the intent explicit.

Notes on ToDo: Changed variable name. This is allocation of space, not actual key_cnt (or key length) of a current page.
CDX-4 nKey = -1 guard assigns nFoundKey from possibly uninitialised slot Minor Risk

At the end of seek() in both cdx.java and idx.java:

if(nKey<0) nKey = 0;
nFoundKey = pgC[nKey];   // pgC[0] may be zero-filled if no page was read

If all visited pages had key_cnt == 0 (an empty index), nKey stays at its loop value and is clamped to 0. Then nFoundKey is set to pgC[0], which was zero-initialised in prepArrs() and has never been written. The array exception is avoided, but nFoundKey contains garbage, and calling code that inspects it on a failed seek may misbehave. A guard like nFoundKey = (key_cnt > 0 ? pgC[nKey] : null) would be cleaner.

Notes on ToDo: Nope. All searched values anyway should be checked. The FoxPro index is like a tool to help finding, not exact guaranteed unique key table. Null is "what exactly"? So what. If no keys then it can be the 0-th too.
CDX-5 Reverse src decoding — correct but entirely undocumented Maintainability

In the LEAF page decode loop:

int src = L;   // 488 — start at TOP of encoded block
// ...
src -= nwb;                                    // walks downward
for(j=0; j<nwb; j++) kb[(dst++)] = bd[src+j];

This is correct per the CDX spec: pack-info bytes fill from the bottom of the 488-byte block upward, and the key bytes fill from the top downward. Without a comment explaining this, any maintainer seeing src = L followed by src -= nwb will spend considerable time verifying it isn't a bug. The entire decode algorithm deserves a two-line comment referencing the spec direction.

Notes on ToDo: If You like it.
CDX-6 cRN is read from the header but never used in bit extraction Maintainability

cRN (number of bits for the record number) is read correctly in readPage():

cRN = read1byte();

But bit extraction uses the derived value bI = 16 - cTC - cDC rather than cRN directly:

int bI = ( 16 - cTC - cDC );   // implicitly assumes cRN == bI
// cRN never referenced again → ends up in dead `notUsedVars` expression

For all VFP-generated CDX files cRN == bI holds, but since cRN is actually available, using it directly (int bI = cRN) would make the assumption explicit and add a natural consistency check. The notUsedVars suppression hack exists precisely to silence this; using cRN meaningfully would remove the need for it.

Notes on ToDo: cRN is important for FoxPro to identify corrupted index. Here is not used in java.
CDX-7 First-key duplicate count zeroed — correct but looks like a guard Clarity 
int dup = (i == 0) ? 0 : ( Q & mDC );  // ← why always 0 for i==0?

The CDX format requires that the very first key on every leaf page carries no duplicate prefix — it is always stored in full. This ternary is therefore semantically correct, not a defensive guard. Without a comment it looks like a protective workaround for an uninitialized Q value, which confuses readers who know the loop is entered fresh. A one-line comment — "// first key is always complete, dup count is meaningless" — removes all ambiguity.

Notes on ToDo: Ignored

IDX IDX Index — Write Path Bugs & Risks

Important note on IDX writing: The readme itself warns against writing IDX files. The review below documents real bugs in the write path (append_key, remove_key). The read path is substantially safer — bugs there are edge-case or cosmetic.
IDX-1 idx write methods have no READ_ONLY guard at all Real Bug

writePage(), writeNumber(), and writeHeaderToEmptyFile() in idx.java write directly to fidx with zero checks. The only protection is that base.isReadOnly() prevents update_key() from being reached — but this is fragile. Consider:

  1. User calls set_order_to_idx() while READ_ONLY = false → fidx opens as "rw".
  2. User later sets READ_ONLY = true.
  3. A code path that bypasses base and calls Idx.replace_key() directly will still write to disk.

Fix: Mirror a boolean readOnly field into idx at open time, and guard writePage():

if(readOnly) return;   // at top of writePage(), writeHeaderToEmptyFile()
Notes on ToDo: User is able to reset READ_ONLY all the time.
It is possible, but READ_ONLY is also internal flag. It can't be 100% fool safe.
Normally user sets it once at the beginning and does not change it till the table is closed.
IDX-2 append_key() page-split leaves fileend inconsistent between the two branches Real Bug (write path)

The two INDEX_PAGES_INSIDE_IDX branches advance fileend differently:

// Branch TRUE (index pages):
fileend += 1024;   // adds 2 × 512 pages
fidx.seek(8); writeNumber(0,4,fileend,true);

// Branch FALSE (flat list):
fileend += 512;    // adds only 1 page
fidx.seek(8); writeNumber(0,4,fileend,true);

In the TRUE branch, two new pages are written at nwR = fileend and nwL = fileend + 512, and then fileend += 1024 is correct. However, the seek-and-write for fileend happens before the new pages are written, meaning a crash mid-write leaves the header pointing beyond any written data. The FALSE branch has the reverse problem — it writes one page but the header update is at the end, which is safer ordering but still increments by only 512 even though left-pointer adjustment has already modified a second page. Reordering: write all pages first, then update the header once, is the safe pattern for both branches.

Notes on ToDo: This is backwards compatibility workaround. FoxPro does it, java doesn't...
Again, use FoxPro databases in read only mode and it will work fast and nice.
IDX-3 remove_key() reads one past array end during shift Real Bug (write path)

The removal shift loop:

for(int i=nKey+1; i<=key_cnt; i++)   // i reaches key_cnt
{
    pgC[i-1] = pgC[i].clone();        // pgC[key_cnt] is out-of-bounds!
    pgR[i-1] = pgR[i];
}

The condition is i <= key_cnt, but valid indices are 0 … key_cnt-1. On the last iteration, pgC[key_cnt] is accessed, which for pgC = new byte[300][100] is within the Java array (no exception), but reads stale data from a previous page load into the last valid slot. The correct bound is i < key_cnt:

for(int i = nKey+1; i < key_cnt; i++) { ... }

IDX-4 cSKey.equals(searchKey) compares byte array references, not contents Real Bug

In base.java → update_key():

if(cSKey==null || (!cSKey.equals(Idx.searchKey)))
    Idx.replace_key();   // always called — equals() on byte[] is identity, not content

byte[].equals() in Java is inherited from Object and performs reference equality, not content comparison. Two different arrays with identical bytes are not .equals(). This means replace_key() is called on every update_key() call regardless of whether the key actually changed, defeating the optimisation entirely and risking unnecessary index corruption on the write path.

Fix:

if(cSKey==null || (!java.util.Arrays.equals(cSKey, Idx.searchKey)))
    Idx.replace_key();

IDX-5 prepareSeekDateTime applies a magic -2 offset to the first key byte Fragile / Undocumented

In idx.java:

public void prepareSeekDateTime( long d )
{
    prepareSeekByLong( d );
    if(d!=0) searchKey[0]-=2;   // "strange but working way"
}

The comment "strange but working way" is honest but dangerous. The T-type (DateTime) key encoding in VFP stores the date as a Julian day number in the high 32 bits and time in milliseconds in the low 32 bits of a double — but the float-to-index-key transform then XORs the sign bit, which shifts the whole byte representation. The -2 likely compensates for the millisecond time component always being non-zero in practice (VFP stores DateTime with at least +1ms). This works only if time is always ≥ 2ms. Document the derivation, or convert DateTime to its two-component form explicitly instead of patching the byte.

BASE base.java — DBF Handling

BASE-1 READ_ONLY flag does not control physical file-open mode Real Bug

This is the most structurally important issue in base.java. The file mode helper:

private String fmode() { return "r" + (READ_ONLY ? "" : "w"); }

…is used at open time. However, cdx_file (which forces READ_ONLY = true inside isReadOnly()) is discovered after the file is already opened. The sequence is:

  1. use() calls fdbf = new RandomAccessFile(dbfname, fmode()) → opens "rw"
  2. getdatabeginptr() reads byte 28 → sets cdx_file = true
  3. From now on isReadOnly() returns true and all writes are logically blocked…
  4. …but fdbf is still physically open for writing at the OS level.

Recommended fix: Peek at byte 28 before the main open, then choose the correct mode:

// Step 1 — probe CDX flag
try (RandomAccessFile probe = new RandomAccessFile(new File(dbfname), "r")) {
    probe.seek(28);
    if(probe.readUnsignedByte() > 0) READ_ONLY = true;
}
// Step 2 — open with correct mode
fdbf = new RandomAccessFile(new File(dbfname), fmode());
Notes on ToDo: Actually user should understand what he does.
Just set READ_ONLY before opening database and reset after close.
BASE-2 isReadOnly() mutates READ_ONLY as a side effect Design Issue

isReadOnly() looks like a pure predicate but secretly sets a public field:

private boolean isReadOnly()
{
    if(cdx_file) READ_ONLY = true;   // ← side effect inside a getter!
    if(READ_ONLY) { throw...; }
    return READ_ONLY;
}

A method named isX() should be a pure query. The mutation of READ_ONLY belongs in the open logic (see BASE-1), not here. This also means every write-guarded method call has the potential to permanently switch the database to read-only mode mid-session in an unexpected way. Separate the two concerns: set READ_ONLY once at open time, and let isReadOnly() just test it.

Notes on ToDo: Same.
BASE-3 All I/O errors are swallowed via e.printStackTrace() and execution continues Design Issue

Virtually every try/catch block in the codebase looks like:

try { fidx.seek(nNode); }
catch (IOException e) { e.printStackTrace(); }   // ← and then execution continues!

After a failed seek(), the following readPage() will read from whatever position the file pointer happens to be in. This silently corrupts the in-memory page buffers. For a read-only database viewer this is survivable — you get a wrong result. For a write path (writePage(), writeNumber()), execution continues and you write to the wrong offset, corrupting the index file. At minimum, critical path I/O (seek, read, write) should set a flag that prevents subsequent operations, or rethrow as an unchecked exception. A simple boolean ioError field checked at the start of each page operation would contain the damage.

Notes on ToDo: The aim is to seek and analyse data, not jump somewhere on error catches.
BASE-4 used() only checks fdbf != null — no check after forced close Minor

used() is defined as:

public boolean used() { return (fdbf != null); }

After use() (close) is called, fdbf is closed but may or may not be set to null depending on the close code path. If an IOException occurs during close, the field might not be nulled, making used() return true for a closed file. The recommended pattern is to always assign fdbf = null in a finally block after close, regardless of exceptions.

Also: setfnames() calls use() at its start ("close previous files"), which is a side-effect of what looks like a name-setting utility. This is non-obvious and can close an open database unexpectedly if setfnames() is called on a used instance.
